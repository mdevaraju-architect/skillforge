# Encryption and Credentials — Shield, Named Credentials, Connected Apps

## Shield Platform Encryption

### Overview

Shield Platform Encryption (SPE) encrypts data at rest using AES-256. Encryption and decryption happen transparently to the Salesforce application layer — users see plaintext values in the UI (subject to FLS). The platform cannot read the data without the tenant secret.

SPE is different from Classic Encryption (a legacy feature that used a fixed, Salesforce-managed key and stored values in a single dedicated encrypted field type).

### Key Components

| Component | Description |
|---|---|
| **Tenant Secret** | A key fragment generated by the customer. Salesforce combines it with an org-specific secret to derive the Data Encryption Key (DEK). |
| **Key Management** | Customer manages the tenant secret. Can be generated in Salesforce or imported (BYOK — Bring Your Own Key). |
| **Data Encryption Key (DEK)** | Derived from tenant secret + platform secret. Used to encrypt field data. Never stored as plaintext. |
| **Encrypted fields** | Specific fields on specific objects, configured in Encryption Policy. Not all field types can be encrypted. |

### Supported Field Types for Encryption

| Field Type | Probabilistic | Deterministic |
|---|---|---|
| Text | Yes | Yes |
| Long Text Area | Yes | No |
| Phone | Yes | Yes |
| Email | Yes | Yes |
| URL | Yes | Yes |
| Date | Yes | No |
| Date/Time | Yes | No |
| Number | No | No |
| Currency | No | No |
| Checkbox | No | No |
| Custom encrypted Text (legacy) | N/A | N/A |

Standard fields that commonly need encryption: SSN, bank account number, credit card number (though storing CC numbers requires PCI scope), date of birth, driver's license number.

### Probabilistic vs Deterministic Encryption

#### Probabilistic Encryption

- Each encryption operation produces a different ciphertext for the same plaintext
- Uses AES-256 in CBC mode with a random IV per encryption
- More secure (ciphertexts are statistically indistinguishable)
- **Cannot be used in:**
  - SOQL `WHERE` clauses (filter)
  - SOQL `ORDER BY`
  - Report filters and groupings
  - List view filters
  - Workflow rule criteria
  - Flow field value conditions
  - Formula field references
  - Duplicate rule matching fields

#### Deterministic Encryption

- Same plaintext always produces the same ciphertext within an org
- Uses AES-256 in SIV (Synthetic IV) mode
- Enables SOQL filtering, ordering, and matching
- Slightly lower theoretical security (ciphertext leaks equality information — an attacker who can observe two ciphertexts can determine if the underlying values are equal)
- Required for any encrypted field used in a WHERE clause, ORDER BY, or formula

**Decision matrix:**

```
Is the field used in a SOQL WHERE, ORDER BY, flow filter,
list view filter, report filter, or formula?
    YES → Use Deterministic encryption
    NO  → Use Probabilistic encryption (stronger)
```

### Setting Up Shield Platform Encryption

1. **Enable Shield:** **Setup → Platform Encryption → Key Management → Generate Tenant Secret**
2. **Configure Encryption Policy:** **Setup → Platform Encryption → Encryption Policy**
   - Select object (e.g. Contact)
   - Select field (e.g. SSN__c)
   - Select encryption scheme: Probabilistic or Deterministic
   - Save
3. **Encrypt existing records:** **Setup → Platform Encryption → Encryption Statistics → Encrypt Existing Records**
   - This is an async batch job
   - Monitor via Encryption Statistics page
   - Large volumes can take hours; plan for this in migration projects

### Impact on SOQL — Silent Failures

The most dangerous aspect of SPE is that probabilistic encryption on a filtered field does not throw an error — it returns 0 results silently.

```apex
// If SSN__c is probabilistic-encrypted, this ALWAYS returns 0 results
// even when a matching record exists
List<Contact> contacts = [
    SELECT Id, Name FROM Contact WHERE SSN__c = '123-45-6789'
];
// contacts.size() == 0 even if a Contact with that SSN exists
```

After applying deterministic encryption to a field, the same query works correctly. This is why the encryption type decision must be made before deployment.

### Key Rotation

- Tenant secrets can be rotated manually or on a schedule
- On rotation: Salesforce re-encrypts all affected field data using the new DEK derived from the new tenant secret
- The old tenant secret is archived (not deleted) so existing data encrypted with the old key can be migrated
- Re-encryption is an async background job
- Do not delete old tenant secrets until re-encryption is complete

### Classic Encryption vs Shield Encryption

| Feature | Classic Encryption | Shield Platform Encryption |
|---|---|---|
| Key management | Salesforce-managed | Customer-managed (tenant secret) |
| Field types | Single encrypted text field type only | Most text-type fields on most objects |
| SOQL filtering | Not supported | Supported with Deterministic |
| Visibility | Masked in UI by default | Visible in UI (controlled by FLS) |
| Cost | Included | Requires Shield add-on license |

Classic Encryption is deprecated in new implementations. Migrate to Shield if Classic fields are in use.

## Named Credentials

### Purpose

Named Credentials store the endpoint URL and authentication settings for external callouts. They decouple configuration from code, enabling the same Apex code to call different endpoints in sandbox vs production.

### Named Credential (Legacy — Pre-API 57.0)

```apex
// Apex callout using a Named Credential
HttpRequest req = new HttpRequest();
req.setEndpoint('callout:MyExternalService/api/v1/data');
req.setMethod('GET');
req.setHeader('Content-Type', 'application/json');

Http http = new Http();
HttpResponse res = http.send(req);
```

The `callout:MyExternalService` prefix tells the platform to look up the Named Credential named `MyExternalService`, substitute the stored endpoint URL and auth headers, and then make the call.

**Setup:** **Setup → Security → Named Credentials → New**

Fields:
- **Label / Name:** The identifier used in `callout:Name`
- **URL:** The base endpoint URL (e.g. `https://api.example.com`)
- **Identity Type:** Named Principal (shared credential) or Per User (each user has their own credential)
- **Authentication Protocol:** No Authentication, Password Authentication (Basic), OAuth 2.0, JWT Token, AWS Signature

For OAuth Named Credentials, the platform handles token refresh automatically — no Apex token management code needed.

### External Credentials (API 57.0+ / Summer 23+)

External Credentials are the modern replacement for Named Credentials' authentication configuration. They separate the endpoint (Named Credential) from the auth config (External Credential), enabling reuse of auth configs across multiple endpoints.

```
Named Credential
    └── URL: https://api.example.com
    └── External Credential: → MyOAuthCredential
            └── Authentication Protocol: OAuth 2.0
            └── Client Id, Client Secret (in vault or org keychain)
            └── Principals: Named Principal or Per User
```

**Apex callout syntax is unchanged:** `callout:MyNamedCredential/path`

**Supported auth protocols in External Credentials:**
- OAuth 2.0 (Client Credentials, JWT Bearer, Auth Code)
- AWS Signature Version 4
- Custom (token injected via formula)

### Never Hardcode Credentials

```apex
// BAD — hardcoded URL and API key
HttpRequest req = new HttpRequest();
req.setEndpoint('https://api.example.com/data');
req.setHeader('X-API-Key', 'abc123secretkey');

// GOOD — Named Credential handles endpoint and auth
req.setEndpoint('callout:MyExternalService/data');
// Platform injects the auth header automatically
```

Hardcoded credentials:
- Are visible to anyone with code access (developers, security reviewers, release managers)
- Cannot be rotated without a code deploy
- Are different between sandbox and production — hardcoded sandbox URL breaks in production
- Violate SOC2, PCI-DSS, and most security frameworks

## Connected Apps

### Purpose

A Connected App allows an external application or service to authenticate to Salesforce using OAuth 2.0 and make API calls on behalf of users (or as a system principal).

### OAuth Flows

| Flow | Use case |
|---|---|
| Authorization Code | User-facing web apps where users grant consent |
| JWT Bearer | Server-to-server (no user interaction, pre-approved) |
| Client Credentials | Machine-to-machine with a dedicated integration user |
| Device | Devices with limited UI (IoT) |
| SAML Bearer | Federated SSO with existing SAML assertions |

### Required Scopes for Common Use Cases

| Need | Required scopes |
|---|---|
| Make REST/SOAP API calls | `api` (or `full`) |
| Access data without user login | `api`, `refresh_token` |
| Read user profile info | `openid`, `profile`, `email` |
| Query Chatter | `chatter_api` |
| Custom scopes only | `custom_permissions` |

**Common mistake:** Including `refresh_token, offline_access` without `api`. The refresh token allows getting new access tokens, but without `api` scope those tokens cannot make API calls. The error `insufficient_scope` is returned at the API call, not at token issuance — which is confusing.

### JWT Bearer Flow Setup

1. Create a Connected App with:
   - OAuth enabled
   - Scopes: `api`, `refresh_token` (or `full`)
   - "Use digital signatures" checked
   - Upload the X.509 certificate (public key) corresponding to your JWT signing key
2. Approve the Connected App for pre-authorization: **Manage → Edit Policies → Permitted Users: Admin approved users are pre-authorized**
3. Assign the permission set or profile that grants access to the Connected App to integration users
4. The integration signs a JWT with the private key, submits to `/services/oauth2/token` with `grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer`
5. Salesforce validates the signature against the uploaded certificate and returns an access token

### Connected App IP Relaxation and Session Policies

**IP Relaxation settings** (on Connected App → Manage → Edit Policies):
- **Enforce IP restrictions** — respects Login IP Ranges from the user's profile
- **Relax IP restrictions with second factor** — bypasses IP restrictions if user completes MFA
- **Relax IP restrictions** — fully bypasses IP restrictions (use carefully in external integrations)

**Session policies** on Connected App:
- **Session Timeout:** Override the org-level session timeout for tokens issued to this Connected App
- **High Assurance required:** Require MFA-level session for this app's tokens

### Consumer Key and Secret Management

- Consumer Key (Client ID): Safe to expose in client-side apps
- Consumer Secret (Client Secret): Must be kept server-side only, never in frontend code or mobile apps
- For public clients (mobile, SPA), use PKCE (Proof Key for Code Exchange) instead of client secret
- Rotate secrets via **Connected App → Manage Consumer Details → Reset Consumer Secret**

## IP Restrictions and Login Hours (Profile-Level)

### Login IP Ranges

Configured on a Profile. Restricts users on that profile to log in only from specified IP ranges. Applies to all login methods including SSO.

```xml
<!-- profiles/StandardUser.profile-meta.xml -->
<loginIpRanges>
    <description>Corporate network</description>
    <endAddress>192.168.1.255</endAddress>
    <startAddress>192.168.1.0</startAddress>
</loginIpRanges>
<loginIpRanges>
    <description>VPN range</description>
    <endAddress>10.0.0.254</endAddress>
    <startAddress>10.0.0.1</startAddress>
</loginIpRanges>
```

**Effect on Connected Apps:** If a Connected App's IP relaxation is "Enforce IP restrictions," users connecting via API from outside the Login IP Range are blocked. Integration users connecting from cloud services (AWS, Azure, CI/CD) must either have their IPs whitelisted or use "Relax IP restrictions" on the Connected App.

### Login Hours

Configured on a Profile. Users on that profile cannot log in outside the specified hours, regardless of authentication method.

**Caution:** Login hours apply to initial authentication. A user already logged in during the allowed window will not be immediately kicked out when the window closes, but their next session validation may fail.
